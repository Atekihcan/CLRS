> Consider a modification to the activity-selection problem in which each activity $$a_i$$ has, in addition to a start and finish time, a value $$v_i$$. The objective is no longer to maximize the number of activities scheduled, but instead to maximize the total value of the activities scheduled. That is, we wish to choose a set $$A$$ of compatible activities such that $$\sum_{a_k \in A} v_k$$ is maximized. Give a polynomial-time algorithm for this problem.

When all activities have equal value, the original greedy approach (earliest finish time) works perfectly. But when activities have different values, the greedy approach fails. A short activity that finishes early might have low value, while a longer activity that blocks others might have very high value. We need to make trade-offs, which calls for dynamic programming.

Think of it as job scheduling where different jobs pay different amounts. Taking a low-paying quick job might prevent you from taking a high-paying long job, so the greedy "finish early" strategy no longer guarantees the best total pay.

### The Algorithm

Sort activities by finish time: $$f_1 \le f_2 \le \cdots \le f_n$$. For each activity $$a_i$$, define $$p(i)$$ as the largest index $$j < i$$ such that activity $$a_j$$ is compatible with $$a_i$$ (i.e., $$f_j \le s_i$$). If no such activity exists, $$p(i) = 0$$.

Let $$\text{OPT}(i)$$ denote the maximum value achievable using only activities $$a_1, a_2, \ldots, a_i$$. The recurrence is:

$$\text{OPT}(i) = \max(\text{OPT}(i-1), v_i + \text{OPT}(p(i)))$$

The first term represents not selecting activity $$a_i$$; the second represents selecting $$a_i$$ and adding its value to the optimal solution for compatible preceding activities.

{% capture code %}
Weighted-Activity-Selection(s, f, v)
    n = s.length
    sort activities by finish time
    compute p[1..n]                       // p[i] = largest j < i with f[j] <= s[i]
    let OPT[0..n] be a new array
    OPT[0] = 0
    for i = 1 to n
        OPT[i] = max(OPT[i-1], v[i] + OPT[p[i]])
    return OPT[n]
{% endcapture %}

### Computing p(i) Efficiently

After sorting by finish time, we can compute each $$p(i)$$ in $$O(\log n)$$ time using binary search. For activity $$a_i$$ with start time $$s_i$$, we search for the largest $$j$$ such that $$f_j \le s_i$$. Since finish times are sorted, this is a standard binary search.

{% capture code %}
Compute-P(s, f)
    n = s.length
    let p[1..n] be a new array
    for i = 1 to n
        // binary search for largest j with f[j] <= s[i]
        lo = 0, hi = i - 1
        while lo < hi
            mid = ceil((lo + hi) / 2)
            if f[mid] <= s[i]
                lo = mid
            else
                hi = mid - 1
        p[i] = lo
    return p
{% endcapture %}

### Reconstructing the Solution

To find which activities to select, we trace back through our decisions:

{% capture code %}
Find-Solution(OPT, p, v, i)
    if i == 0
        return empty set
    else if OPT[i-1] >= v[i] + OPT[p[i]]
        return Find-Solution(OPT, p, v, i-1)
    else
        return {a_i} ∪ Find-Solution(OPT, p, v, p[i])
{% endcapture %}

### Running Time Analysis

- Sorting: $$O(n \log n)$$
- Computing all $$p(i)$$ values: $$O(n \log n)$$ ($$n$$ binary searches)
- Dynamic programming: $$O(n)$$
- Reconstructing solution: $$O(n)$$

Total: $$O(n \log n)$$

{% capture note %}
This problem is known as the [weighted job scheduling](https://en.wikipedia.org/wiki/Interval_scheduling#Weighted){:target="_blank"} or weighted interval scheduling problem. The unweighted version permits a greedy solution, but adding weights forces us to consider trade-offs between "finishing early" and "high value," which requires dynamic programming. This illustrates how small problem modifications can fundamentally change the algorithmic approach.
