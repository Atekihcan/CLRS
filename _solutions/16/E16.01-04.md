> Suppose that we have a set of activities to schedule among a large number of lecture halls, where any activity can take place in any lecture hall. We wish to schedule all the activities using as few lecture halls as possible. Give an efficient greedy algorithm to determine which activity should use which lecture hall.
>
> (This problem is also known as the interval-graph coloring problem. We can create an interval graph whose vertices are the given activities and whose edges connect incompatible activities. The smallest number of colors required to color every vertex so that no two adjacent vertices have the same color corresponds to finding the fewest lecture halls needed to schedule all of the given activities.)

Instead of selecting a maximum subset of compatible activities for one room, we now need to assign all activities to rooms, minimizing the number of rooms used. Think of it as: how many parallel tracks do we need at a conference to accommodate all talks without conflicts?

The key insight is that the minimum number of rooms needed equals the maximum number of activities that overlap at any single point in time. If three activities all include time $$t$$, we need at least three rooms. The greedy algorithm achieves this lower bound by assigning each activity to any available room, opening a new room only when necessary.

### The Algorithm

Sort all activities by start time. Process them in this order, assigning each activity to any room that is free (i.e., whose current activity has finished). If no room is free, open a new room.

{% capture code %}
Schedule-Lecture-Halls(s, f)
    n = s.length
    sort activities by start time
    rooms = empty list                    // each room tracks when it becomes free
    assignment = new array[1..n]
    for i = 1 to n
        assigned = false
        for each room r in rooms
            if room r is free at time s[i]  // room's current activity finished
                assign activity i to room r
                update room r's free time to f[i]
                assignment[i] = r
                assigned = true
                break
        if not assigned
            create new room r
            assign activity i to room r
            update room r's free time to f[i]
            assignment[i] = r
            add r to rooms
    return assignment, |rooms|
{% endcapture %}

### Efficient Implementation Using a Min-Heap

To efficiently find a free room, we maintain a min-heap of rooms keyed by their "free time" (when their current activity ends). For each new activity, we check if the room with the earliest free time is available. If so, we use it and update the heap; otherwise, we add a new room.

{% capture code %}
Schedule-Lecture-Halls-Efficient(s, f)
    n = s.length
    sort activities by start time to get permutation π
    H = empty min-heap                    // keyed by free time
    num_rooms = 0
    assignment = new array[1..n]
    for i = 1 to n
        j = π[i]                          // activity j has i-th earliest start time
        if H is not empty and min(H).free_time <= s[j]
            room = Extract-Min(H)         // reuse this room
            room.free_time = f[j]
            Insert(H, room)
            assignment[j] = room.id
        else
            num_rooms = num_rooms + 1     // need a new room
            new_room = (id: num_rooms, free_time: f[j])
            Insert(H, new_room)
            assignment[j] = num_rooms
    return assignment, num_rooms
{% endcapture %}

### Correctness

The algorithm is correct because it uses a new room only when all existing rooms are occupied. At any point, if we have $$k$$ rooms in use and need to open room $$k+1$$, it means $$k$$ activities are currently ongoing and overlap with the new activity. Thus, $$k+1$$ activities overlap at some moment, proving we need at least $$k+1$$ rooms.

More formally, the number of rooms used equals the maximum "depth" of overlapping activities, which is a lower bound on the number of rooms required. Since the algorithm achieves this lower bound, it is optimal.

### Running Time

Sorting takes $$O(n \log n)$$ time. Each of the $$n$$ activities requires one heap operation (either $$\textsc{Extract-Min}$$ followed by $$\textsc{Insert}$$, or just $$\textsc{Insert}$$), each taking $$O(\log n)$$ time. The total running time is $$O(n \log n)$$.

{% capture note %}
This problem is equivalent to finding the minimum number of colors needed to color an [interval graph](https://en.wikipedia.org/wiki/Interval_graph){:target="_blank"}. For interval graphs, the chromatic number equals the clique number (maximum number of pairwise-overlapping intervals), and both can be computed in linear time after sorting. This is special; for general graphs, graph coloring is NP-hard.
