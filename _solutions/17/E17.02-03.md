> Suppose we wish not only to increment a counter but also to reset it to zero (i.e., make all bits in it 0). Counting the time to examine or modify a bit as $$\Theta(1)$$, show how to implement a counter as an array of bits so that any sequence of $$n$$ $$\textsc{Increment}$$ and $$\textsc{Reset}$$ operations takes time $$O(n)$$ on an initially zero counter. (Hint: Keep a pointer to the high-order 1.)

The challenge with $$\textsc{Reset}$$ is that naively clearing all $$k$$ bits would cost $$\Theta(k)$$, potentially making a sequence of $$n$$ operations cost $$\Theta(nk)$$. The hint suggests maintaining a pointer to track the highest-order 1-bit, so we only clear bits that are actually set.

Picture a row of light switches where you need to occasionally turn them all off. Instead of walking down the entire row checking each switch, you keep track of how far down the row any switch is on. When resetting, you only need to flip switches up to that point.

We maintain a counter array $$A[0..k-1]$$ plus a pointer $$h$$ that tracks the index of the highest-order bit that is (or might be) set to 1. Initially, $$h = -1$$ (no bits are set).

{% capture code %}
Increment(A, h)
    i = 0
    while i < A.length and A[i] == 1
        A[i] = 0
        i = i + 1
    if i < A.length
        A[i] = 1
        if i > h
            h = i
{% endcapture %}

{% capture code %}
Reset(A, h)
    for i = 0 to h
        A[i] = 0
    h = -1

Now we analyze using the accounting method. Let us charge:

- $$\textsc{Increment}$$: amortized cost of 2
- $$\textsc{Reset}$$: amortized cost of 0

For $$\textsc{Increment}$$: When we set a bit from 0 to 1 (line 5), we use 1 dollar for the actual operation and place 1 dollar on that bit as credit. When we reset a bit from 1 to 0 (line 3), we use the credit stored on that bit to pay for it.

For $$\textsc{Reset}$$: Each bit we clear from 1 to 0 has a dollar of credit on it (deposited when it was last set to 1). We use this credit to pay for clearing the bit. We also reset $$h$$ to $$-1$$ at cost $$O(1)$$.

To see why the credit is always nonnegative: every bit that is currently 1 has exactly 1 dollar of credit on it. The $$\textsc{Increment}$$ operation sets at most one bit to 1 and deposits a dollar. Both $$\textsc{Increment}$$ (resetting bits in the while loop) and $$\textsc{Reset}$$ (clearing bits to $$h$$) only clear bits that are currently 1, using up their stored credit.

The total amortized cost for $$n$$ operations is at most $$2n$$ (if every operation were an $$\textsc{Increment}$$), which is $$O(n)$$. Since the total amortized cost is an upper bound on the total actual cost, any sequence of $$n$$ $$\textsc{Increment}$$ and $$\textsc{Reset}$$ operations takes $$O(n)$$ time.

{% capture note %}
The pointer $$h$$ is essential for making $$\textsc{Reset}$$ efficient. Without it, we'd have to scan all $$k$$ bits to find which ones to clear. With $$h$$, we only iterate up to the highest set bit. But interestingly, the accounting analysis doesn't directly use $$h$$; the pointer just ensures we don't touch bits that are already 0 (which would have no credit).
{% endcapture %}
