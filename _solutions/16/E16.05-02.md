> Show how to use property 2 of Lemma 16.12 to determine in time $$O(\lvert A \rvert)$$ whether or not a given set $$A$$ of tasks is independent.

Property 2 of Lemma 16.12 states that a set of tasks $$A$$ is independent if and only if for all $$t = 0, 1, 2, \ldots, n$$, we have $$N_t(A) \le t$$, where $$N_t(A)$$ is the number of tasks in $$A$$ whose deadline is $$t$$ or earlier.

A naive approach would check this condition for every value of $$t$$ from 1 to $$n$$, but that could take $$O(n)$$ time even if $$\lvert A \rvert$$ is much smaller than $$n$$. The key insight is that we only need to check the condition at deadlines that actually appear in $$A$$, since those are the only points where $$N_t(A)$$ changes.

**Algorithm**

1. Sort the tasks in $$A$$ by their deadlines in increasing order. This takes $$O(\lvert A \rvert \log \lvert A \rvert)$$ time.

2. Scan through the sorted tasks, keeping a counter $$k$$ of how many tasks we have seen so far. For each task $$a_i$$ encountered:
   - Increment $$k$$
   - If $$k > d_i$$ (more tasks seen than the current deadline allows), return **not independent**

3. If we complete the scan without violating the condition, return **independent**.

The scan itself takes $$O(\lvert A \rvert)$$ time. With the sort, the total time is $$O(\lvert A \rvert \log \lvert A \rvert)$$.

**Achieving $$O(\lvert A \rvert)$$ Time**

To achieve true $$O(\lvert A \rvert)$$ time, we can use **counting sort** if deadlines are integers in a bounded range, or we can observe that if the tasks have already been sorted (which is the case in the $$\textsc{Greedy}$$ algorithm where we process tasks in order of decreasing penalty), we can maintain the sorted order incrementally.

Here is the $$O(\lvert A \rvert)$$ approach assuming tasks can be sorted by deadline in linear time:

1. Create an array $$\text{count}[1 \ldots m]$$ initialized to 0, where $$m = \lvert A \rvert$$.

2. For each task $$a_i \in A$$:
   - Let $$d = \min(d_i, m)$$ (deadlines larger than $$m$$ are effectively capped at $$m$$ since we only have $$m$$ tasks)
   - Increment $$\text{count}[d]$$

3. Compute prefix sums: set $$N_t = \sum_{j=1}^{t} \text{count}[j]$$ for $$t = 1, \ldots, m$$.

4. For each $$t$$ from 1 to $$m$$:
   - If $$N_t > t$$, return **not independent**

5. Return **independent**.

Steps 2-4 each take $$O(\lvert A \rvert)$$ time.

**Why This Works**

The condition $$N_t(A) \le t$$ needs to hold for all $$t$$. However:
- For $$t > m = \lvert A \rvert$$, we automatically have $$N_t(A) \le \lvert A \rvert \le t$$, so these constraints are always satisfied.
- For $$t \le m$$, deadlines $$d_i > m$$ contribute to $$N_t(A)$$ only when $$t \ge d_i$$, but since $$d_i > m \ge t$$, they do not contribute at all for the relevant range.

By capping deadlines at $$m$$, we ensure tasks with "generous" deadlines (larger than $$\lvert A \rvert$$) do not affect the outcome, while still correctly counting tasks with tight deadlines.

This independence check is the key subroutine that makes the greedy scheduling algorithm work. Since the greedy algorithm performs $$O(n)$$ independence checks (one for each task considered), and each check takes $$O(\lvert A \rvert)$$ time where $$\lvert A \rvert \le n$$, the overall algorithm runs in $$O(n^2)$$ time.
