> Suppose that a data file contains a sequence of 8-bit characters such that all 256 characters are about equally common: the maximum character frequency is less than twice the minimum character frequency. Prove that Huffman coding in this case is no more efficient than using an ordinary 8-bit fixed-length code.

We need to show that when character frequencies are nearly uniform (maximum is less than twice the minimum), every character in the optimal Huffman code has a codeword of length exactly 8.

Let $$f_{\min}$$ and $$f_{\max}$$ be the minimum and maximum frequencies among the 256 characters. We're given that $$f_{\max} < 2 \cdot f_{\min}$$.

The key observation involves a property of Huffman trees: in any optimal prefix code tree, if a character $$c$$ has a codeword of length $$\ell$$, then there must be at least two characters at depth $$\ell$$ (they would be siblings in the tree), and their combined frequency is at least as large as the frequency of any character at depth $$\ell - 1$$.

Suppose for contradiction that some character $$x$$ has a codeword of length 9 or more in the Huffman tree. Then $$x$$ has a sibling $$y$$ at the same depth (since the tree is full by Exercise 16.3-2). Their parent $$p$$ has frequency $$f_x + f_y \ge 2 \cdot f_{\min}$$.

Now consider the character $$z$$ with maximum frequency $$f_{\max}$$. We have:

$$f_{\max} < 2 \cdot f_{\min} \le f_x + f_y$$

This means the internal node $$p$$ (parent of $$x$$ and $$y$$) has frequency strictly greater than $$f_{\max}$$.

Think about how the Huffman algorithm proceeds. It repeatedly extracts the two minimum-frequency nodes and merges them. When we merge $$x$$ and $$y$$ to create $$p$$, the node $$p$$ has frequency greater than $$f_{\max}$$.

For $$p$$ to have depth greater than or equal to 8, it must be merged with another node at some point before we reach a tree with 256 leaves at depths at most 8. But since $$p.\text{freq} > f_{\max} \ge f_c$$ for every original character $$c$$, and we always merge minimum-frequency nodes, $$p$$ will not be selected for merging until all individual characters have been grouped into subtrees.

Here's the critical insight: a full binary tree with 256 leaves has depth exactly 8 if and only if it's a complete binary tree (all leaves at depth 8). For the Huffman tree to have any leaf at depth 9 or greater, we would need:
- At least one leaf at depth $$\ge 9$$
- This leaf has a sibling, both under a common parent at depth $$\ge 8$$

But we just showed that any parent of two minimum-frequency characters has frequency $$\ge 2f_{\min} > f_{\max}$$. This parent would not be selected for further merging while any singleton character remains, so it becomes part of the "top" of the tree, not deeper in.

Let's prove it more directly: In a Huffman tree with 256 characters, a leaf at depth $$d$$ means there are $$2^d$$ or more nodes at depth $$d$$ in a complete binary tree sense. With frequencies nearly equal, the Huffman algorithm essentially builds a balanced tree because no character has frequency large enough to "float up" significantly higher than others. Since $$256 = 2^8$$, a balanced tree with 256 leaves has all leaves at depth 8.

More formally, with all frequencies between $$f_{\min}$$ and $$2f_{\min}$$, after any sequence of merges, the resulting internal node frequencies are also bounded. The tree remains balanced enough that no branch extends beyond depth 8.

Therefore, every character has a codeword of length exactly 8, which is the same as a fixed-length code. The Huffman code provides no compression benefit.

{% capture note %}
This result makes intuitive sense: compression works by exploiting non-uniformity in the data. If all characters appear with roughly equal frequency, there's no redundancy to exploit, and variable-length coding provides no advantage over fixed-length coding.
