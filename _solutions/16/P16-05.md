> ***Off-line caching***
>
> Modern computers use a cache to store a small amount of data in a fast memory. Even though a program may access large amounts of data, by storing a small subset of the main memory in the cache (a small but faster memory) overall access time can greatly decrease. When a computer program executes, it makes a sequence $$\langle r_1, r_2, \ldots, r_n \rangle$$ of $$n$$ memory requests, where each request is for a particular data element. For example, a program that accesses 4 distinct elements $$\{a, b, c, d\}$$ might make the sequence of requests $$\langle d, b, d, b, d, a, c, d, b, a, c, b \rangle$$. Let $$k$$ be the size of the cache. When the cache contains $$k$$ elements and the program requests the $$(k+1)$$st element, the system must decide, for this and each subsequent request, which $$k$$ elements to keep in the cache. More precisely, for each request $$r_i$$, the cache-management algorithm checks whether element $$r_i$$ is already in the cache. If it is, then we have a cache hit; otherwise, we have a cache miss. Upon a cache miss, the system retrieves $$r_i$$ from the main memory, and the cache-management algorithm must decide whether to keep $$r_i$$ in the cache. If it decides to keep $$r_i$$ and the cache already holds $$k$$ elements, then it must evict one element to make room for $$r_i$$. The cache-management algorithm evicts data with the goal of minimizing the number of cache misses over the entire sequence of requests.
>
> Typically, caching is an on-line problem. That is, we have to make decisions about which data to keep in the cache without knowing the future requests. Here, however, we consider the off-line version of this problem, in which we are given in advance the entire sequence of $$n$$ requests and the cache size $$k$$, and we wish to minimize the total number of cache misses.
>
> We can solve this off-line problem by a greedy strategy called furthest-in-future, which chooses to evict the item in the cache whose next access in the request sequence comes furthest in the future.
>
> 1. Write pseudocode for a cache manager that uses the furthest-in-future strategy. The input should be a sequence $$\langle r_1, r_2, \ldots, r_n \rangle$$ of requests and a cache size $$k$$, and the output should be a sequence of decisions about which data element (if any) to evict upon each request. What is the running time of your algorithm?
> 2. Show that the off-line caching problem exhibits optimal substructure.
> 3. Prove that furthest-in-future produces the minimum possible number of cache misses.

The furthest-in-future algorithm, also known as [Belady's algorithm](https://en.wikipedia.org/wiki/B%C3%A9l%C3%A1dy%27s_algorithm){:target="_blank"}, is the optimal offline cache replacement policy. The intuition is straightforward: if we must evict something, evict the item we won't need for the longest time. This postpones the "cost" of the eviction as far as possible.

### A

{% capture code %}
Furthest-In-Future(R, n, k)
    // R[1..n] = sequence of requests
    // k = cache size
    // Returns: eviction[1..n] where eviction[i] = element evicted at step i, or nil

    let cache be an empty set
    let eviction[1..n] = nil for all i

    // Precompute next occurrence of each element after each position
    let next[1..n] = infinity for all i
    let last-seen be a dictionary

    // Scan backwards to compute next occurrence
    for i = n downto 1
        if R[i] is in last-seen
            next[i] = last-seen[R[i]]
        else
            next[i] = infinity
        last-seen[R[i]] = i

    for i = 1 to n
        if R[i] is in cache
            // Cache hit, no action needed
            continue
        // Cache miss
        if |cache| < k
            // Cache not full, just add the element
            insert R[i] into cache
        else
            // Cache full, must evict
            // Find element in cache with furthest next use
            furthest = -1
            victim = nil
            for each element e in cache
                // Find next occurrence of e after position i
                next-use = find-next-use(e, i, R, n)
                if next-use > furthest
                    furthest = next-use
                    victim = e
            eviction[i] = victim
            remove victim from cache
            insert R[i] into cache

    return eviction
{% endcapture %}

The helper function $$\textsc{Find-Next-Use}(e, i, R, n)$$ finds the next position after $$i$$ where element $$e$$ is requested. If $$e$$ is never requested again, it returns $$\infty$$.

For an efficient implementation, we precompute the next occurrence of each element from each position. We can do this in $$O(n)$$ time by scanning the request sequence backwards and maintaining for each distinct element the position of its last seen occurrence.

{% include ads.html %}

**Running Time Analysis:**

- Preprocessing (computing next occurrences): $$O(n)$$
- Main loop: $$n$$ iterations
  - Cache lookup: $$O(k)$$ or $$O(1)$$ with a hash set
  - Finding the victim: requires scanning all $$k$$ cache elements and looking up each one's next occurrence
  - With precomputed next occurrences stored in an array indexed by position, the lookup is $$O(1)$$
  - But we need to find which cache element has the furthest next use after the current position

A naive implementation takes $$O(k)$$ per miss to find the victim, giving $$O(nk)$$ total time.

With a more sophisticated data structure (a balanced BST or priority queue keyed by next-use time, updated as we process requests), we can achieve $$O(n \log k)$$ time. Each element enters and leaves the cache at most once between consecutive accesses, and we have at most $$n$$ requests, giving $$O(n)$$ cache operations each taking $$O(\log k)$$ time.

The simplest implementation runs in $$O(nk)$$ time. The optimized version runs in $$O(n \log k)$$ time.

### B

The off-line caching problem exhibits optimal substructure as follows:

Consider an optimal solution that processes requests $$r_1, r_2, \ldots, r_n$$. After processing request $$r_i$$, suppose the cache contains elements $$C_i$$ and we have incurred $$m_i$$ cache misses so far. The remaining problem is to process requests $$r_{i+1}, \ldots, r_n$$ starting with cache contents $$C_i$$.

**Claim:** The decisions made for $$r_{i+1}, \ldots, r_n$$ in the optimal solution must themselves be optimal for the subproblem of minimizing cache misses over $$r_{i+1}, \ldots, r_n$$ starting with cache $$C_i$$.

**Proof (by cut-and-paste):** Suppose the decisions for $$r_{i+1}, \ldots, r_n$$ incur $$m'$$ cache misses, but there exists an alternative strategy for this subproblem that incurs only $$m'' < m'$$ misses. Then we could substitute this better strategy, achieving a total of $$m_i + m''$$ misses instead of $$m_i + m'$$, contradicting the optimality of our original solution.

{% include ads.html %}

This optimal substructure is slightly different from typical DP problems because the subproblem is defined by both the remaining requests AND the current cache state. The state space is large (the cache can contain any $$k$$ of the distinct elements), which is why a greedy approach that avoids explicitly exploring all states is valuable.

### C

We prove that the furthest-in-future (FIF) strategy produces the minimum number of cache misses using an exchange argument.

Let $$S$$ be the schedule produced by FIF, and let $$S^*$$ be any optimal schedule. We will show that $$S$$ has no more cache misses than $$S^*$$.

We transform $$S^*$$ into $$S$$ step by step, ensuring that the number of cache misses never increases. At each step, we consider the first request where $$S$$ and $$S^*$$ make different decisions.

**Case 1: Cache hit in both.** No difference in eviction decisions, move to the next request.

**Case 2: Cache miss in both, same eviction choice.** Move to the next request.

**Case 3: Cache miss in both, different eviction choice.** Suppose at request $$r_i$$, both strategies have a cache miss and must evict an element to bring in $$r_i$$. Let $$S$$ evict element $$f$$ (the furthest-in-future element) and $$S^*$$ evict element $$g \neq f$$.

Since FIF chose $$f$$, we know that $$f$$ is accessed later than $$g$$ (or never accessed again). Let $$t$$ be the first time after $$i$$ where $$f$$ or $$g$$ is requested.

- If neither $$f$$ nor $$g$$ is ever requested again: Both strategies have the same number of future misses for these elements.

- If $$g$$ is requested at time $$t$$ (before $$f$$'s next request):
  - In $$S^*$$: $$g$$ was evicted, so $$S^*$$ has a cache miss at time $$t$$.
  - In $$S$$: $$g$$ is still in cache, so $$S$$ has a cache hit at time $$t$$.

  We can modify $$S^*$$ to evict $$f$$ instead of $$g$$ at time $$i$$. This saves a miss at time $$t$$ (now $$g$$ is in cache). At some later point when $$f$$ is needed, $$S^*$$ might incur an extra miss, but this is balanced by the miss we saved. The key insight is that by the time $$f$$ is needed, the cache states of the modified $$S^*$$ and $$S$$ will be identical.

- If $$f$$ is requested at time $$t$$ first:
  - This contradicts our assumption that FIF evicts the element with the furthest next use.

By repeatedly applying this transformation, we convert $$S^*$$ into $$S$$ without increasing the number of cache misses. Therefore, $$S$$ (the FIF schedule) has at most as many cache misses as any optimal schedule $$S^*$$. Since $$S^*$$ was optimal, $$S$$ must also be optimal.

This proves that the furthest-in-future strategy produces the minimum possible number of cache misses.
