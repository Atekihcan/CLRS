> Give pseudocode to reconstruct an LCS from the completed $$c$$ table and the original sequences $$X = \langle x_1, x_2, \ldots, x_m \rangle$$ and $$Y = \langle y_1, y_2, \ldots, y_n \rangle$$ in $$O(m + n)$$ time, without using the $$b$$ table.

The key insight is that we don't actually need the $$b$$ table to trace back through the $$c$$ table. At any position $$c[i, j]$$, we can determine which subproblem we came from by examining the values in the $$c$$ table and comparing $$x_i$$ with $$y_j$$.

Think of it like retracing your steps through a maze. At each cell, we can figure out how we got there by looking at the neighboring cells and remembering the rules we followed to fill in the table originally. If $$x_i = y_j$$, we must have come from the diagonal (and this character is part of the LCS). Otherwise, we came from whichever neighbor has the larger value.

{% capture code %}
Print-LCS(c, X, Y, i, j)
    if i == 0 or j == 0
        return
    if x_i == y_j
        Print-LCS(c, X, Y, i - 1, j - 1)
        print x_i
    elseif c[i - 1, j] >= c[i, j - 1]
        Print-LCS(c, X, Y, i - 1, j)
    else
        Print-LCS(c, X, Y, i, j - 1)
{% endcapture %}

The initial call is $$\textsc{Print-LCS}(c, X, Y, m, n)$$ where $$m = X.length$$ and $$n = Y.length$$.

{% include ads.html %}

The running time is $$O(m + n)$$ because each recursive call decreases either $$i$$ or $$j$$ (or both) by 1. Since $$i$$ starts at $$m$$ and $$j$$ starts at $$n$$, and both can only decrease until they reach 0, the total number of recursive calls is at most $$m + n$$.

{% capture note %}
By checking whether $$x_i = y_j$$ first, we naturally handle the case where the current element is part of the LCS. When there's a match, we know the value $$c[i, j]$$ came from $$c[i-1, j-1] + 1$$, so we print the matching character and move diagonally. When there's no match, we simply follow the path of maximum value, just as we would have done using the $$b$$ table.
{% endcapture %}
