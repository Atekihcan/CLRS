> ***Inventory planning***
>
> The Rinky Dink Company makes machines that resurface ice rinks. The demand for such products varies from month to month, and so the company needs to develop a strategy to plan its manufacturing given the fluctuating, but predictable, demand. The company wishes to design a plan for the next $$n$$ months. For each month $$i$$, the company knows the demand $$d_i$$, that is, the number of machines that it will sell. Let $$D = \sum_{i=1}^{n} d_i$$ be the total demand over the next $$n$$ months. The company keeps a full-time staff who provide labor to manufacture up to $$m$$ machines per month. If the company needs to make more than $$m$$ machines in a given month, it can hire additional, part-time labor, at a cost that works out to $$c$$ dollars per machine. Furthermore, if, at the end of a month, the company is holding any unsold machines, it must pay inventory costs. The cost for holding $$j$$ machines is given as a function $$h(j)$$ for $$j = 1, 2, \ldots, D$$, where $$h(j) \geq 0$$ for $$1 \leq j \leq D$$ and $$h(j) \leq h(j+1)$$ for $$1 \leq j \leq D - 1$$.
>
> Give an algorithm that calculates a plan for the company that minimizes its costs while fulfilling all the demand. The running time should be polynomial in $$n$$ and $$D$$.

This is a classic production planning problem. The company faces a tradeoff: build machines early (when you might have spare capacity) and pay inventory costs, or build them late (when demand spikes) and pay for part-time labor. Finding the right balance requires looking at the entire planning horizon.

Think of it like packing for a trip where luggage fees change daily. Sometimes it's cheaper to carry extra weight early (pay holding costs) to avoid premium prices later (part-time labor costs).

### Characterizing the State

At any point in time, what matters is: which month are we in, and how many machines do we have in inventory? The number of machines we'll produce this month depends on current inventory, this month's demand, and future considerations.

Let $$\text{cost}[i][j]$$ be the minimum cost to satisfy demand for months $$i$$ through $$n$$, given that we start month $$i$$ with $$j$$ machines in inventory.

### The Recurrence

At the start of month $$i$$ with inventory $$j$$, we must decide how many machines $$p$$ to produce. The constraints are:
- We must meet demand: $$j + p \geq d_i$$
- Production must be non-negative: $$p \geq 0$$
- Reasonable upper bound: we never need more than total remaining demand in inventory

The cost of producing $$p$$ machines:
- If $$p \leq m$$: $$0$$ (covered by full-time staff)
- If $$p > m$$: $$c \cdot (p - m)$$ (part-time labor for excess)

After production and sales, our new inventory is $$j' = j + p - d_i$$, incurring holding cost $$h(j')$$ if $$j' > 0$$.

The recurrence is:

$$\text{cost}[i][j] = \min_{p} \left\{ \text{prod\_cost}(p) + \text{hold\_cost}(j + p - d_i) + \text{cost}[i+1][j + p - d_i] \right\}$$

where the minimum is over all valid production amounts $$p$$.

### The Algorithm

{% capture code %}
Inventory-Planning(d, n, m, c, h)
    // D = total demand
    D = 0
    for i = 1 to n
        D = D + d[i]

    // cost[i][j] = min cost for months i..n starting with inventory j
    let cost[1..n+1, 0..D] be a new table
    let produce[1..n, 0..D] be a new table  // for reconstruction

    // Base case: after month n, no more costs
    for j = 0 to D
        cost[n+1][j] = 0

    // Fill table backwards from month n to month 1
    for i = n downto 1
        // Compute remaining demand from month i onwards
        remaining = 0
        for k = i to n
            remaining = remaining + d[k]

        for j = 0 to D
            cost[i][j] = infinity

            // Minimum production: must meet this month's demand
            min_prod = max(0, d[i] - j)

            // Maximum production: no point having more than remaining demand
            max_prod = remaining - j

            for p = min_prod to max_prod
                // Production cost
                if p <= m
                    prod_cost = 0
                else
                    prod_cost = c * (p - m)

                // New inventory after this month
                new_inv = j + p - d[i]

                // Holding cost
                if new_inv > 0
                    hold_cost = h[new_inv]
                else
                    hold_cost = 0

                // Total cost
                total = prod_cost + hold_cost + cost[i+1][new_inv]

                if total < cost[i][j]
                    cost[i][j] = total
                    produce[i][j] = p

    return cost[1][0], produce

### Complexity Analysis

**State space:** We have $$n$$ months and at most $$D$$ possible inventory levels. However, the inventory at month $$i$$ is bounded by the remaining demand from months $$i$$ to $$n$$. The total number of states is $$O(n \cdot D)$$.

**Transitions:** For each state, we consider at most $$D$$ production choices (more precisely, bounded by remaining demand).

**Time complexity:** $$O(n \cdot D^2)$$

This is polynomial in $$n$$ and $$D$$ as required.

**Space complexity:** $$O(n \cdot D)$$ for the tables.

### Why This Works

The key insight is that inventory level captures everything we need to know about the past. It doesn't matter how we got to a particular inventory level; what matters for future decisions is simply how many machines we currently have.

This memoryless property gives us optimal substructure: the best way to handle months $$i$$ through $$n$$ starting with $$j$$ machines doesn't depend on how we accumulated those $$j$$ machines.

{% capture note %}
The non-decreasing property of $$h(j)$$ (holding costs increase with inventory) is natural: storing more machines costs more. This property also ensures we won't build arbitrarily large inventories, which helps bound our state space.
