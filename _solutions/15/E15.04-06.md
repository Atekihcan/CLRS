> $$\star$$ Give an $$O(n \lg n)$$-time algorithm to find the longest monotonically increasing subsequence of a sequence of $$n$$ numbers. (*Hint: Observe that the last element of a candidate subsequence of length $$i$$ is at least as large as the last element of a candidate subsequence of length $$i - 1$$. Maintain candidate subsequences by linking them through the input sequence.*)

The key observation in the hint is profound: if we have found an increasing subsequence of length $$k$$ ending with some value $$v$$, then any increasing subsequence of length $$k - 1$$ must end with a value less than $$v$$. This means we can maintain an array where position $$i$$ stores the smallest ending value among all increasing subsequences of length $$i$$.

Consider processing elements one at a time. For each new element $$x$$:
- If $$x$$ is larger than all ending values we've seen, we can extend our longest subsequence
- Otherwise, $$x$$ can potentially improve (i.e., reduce) the ending value for some subsequence length

Let $$T[k]$$ store the **smallest** ending value of any increasing subsequence of length $$k$$ found so far. The crucial property is that $$T$$ is always sorted in increasing order. This is because a longer increasing subsequence must end with a value at least as large as a shorter one (you can't extend an increasing subsequence with a smaller element).

Since $$T$$ is sorted, we can use binary search to find where each new element belongs, giving us $$O(\log n)$$ per element and $$O(n \log n)$$ overall.

{% capture code %}
LIS-Fast(A)
    n = A.length
    let T[1..n] be a new array
    let P[1..n] be a new array  // P[i] = predecessor of A[i] in LIS
    let B[1..n] be a new array  // B[k] = index in A of element stored at T[k]
    length = 0
    for i = 1 to n
        // Binary search for largest j such that T[j] < A[i]
        lo = 1
        hi = length
        while lo <= hi
            mid = floor((lo + hi) / 2)
            if T[mid] < A[i]
                lo = mid + 1
            else
                hi = mid - 1
        // lo is now the position where A[i] should go
        T[lo] = A[i]
        B[lo] = i
        if lo > 1
            P[i] = B[lo - 1]
        else
            P[i] = NIL
        if lo > length
            length = length + 1
    // Reconstruct the LIS
    Print-LIS-Reverse(A, P, B[length])
    return length
{% endcapture %}

Let's trace through an example with $$A = \langle 3, 1, 4, 1, 5, 9, 2, 6 \rangle$$:

| Step | Element | $$T$$ after update | Action |
|------|---------|-------------------|--------|
| 1 | 3 | $$\langle 3 \rangle$$ | Start new LIS of length 1 |
| 2 | 1 | $$\langle 1 \rangle$$ | Replace 3 with 1 (better ending for length 1) |
| 3 | 4 | $$\langle 1, 4 \rangle$$ | Extend to length 2 |
| 4 | 1 | $$\langle 1, 4 \rangle$$ | 1 equals $$T[1]$$, no change needed |
| 5 | 5 | $$\langle 1, 4, 5 \rangle$$ | Extend to length 3 |
| 6 | 9 | $$\langle 1, 4, 5, 9 \rangle$$ | Extend to length 4 |
| 7 | 2 | $$\langle 1, 2, 5, 9 \rangle$$ | Replace 4 with 2 (better ending for length 2) |
| 8 | 6 | $$\langle 1, 2, 5, 6 \rangle$$ | Replace 9 with 6 (better ending for length 4) |

The final LIS length is 4, with one possible LIS being $$\langle 1, 4, 5, 9 \rangle$$ or $$\langle 1, 2, 5, 6 \rangle$$.

The running time is $$O(n \log n)$$:
- We process each of the $$n$$ elements exactly once
- For each element, we perform a binary search on $$T$$, which has at most $$n$$ elements
- Each binary search takes $$O(\log n)$$ time
- Therefore, total time is $$O(n \log n)$$

{% capture note %}
The array $$T$$ does **not** contain an actual increasing subsequence! It contains the best possible ending values for subsequences of each length. For example, after processing $$\langle 3, 1, 4, 1, 5, 9, 2, 6 \rangle$$, we have $$T = \langle 1, 2, 5, 6 \rangle$$, but $$\langle 1, 2, 5, 6 \rangle$$ isn't a subsequence of the input (the elements don't appear in that order). To reconstruct an actual LIS, we need the predecessor pointers stored in $$P$$ and the index tracking array $$B$$.
{% endcapture %}
{% include aside.html title='A Common Misconception' %}

This algorithm is sometimes called the "patience sorting" algorithm because it resembles the card game Patience (Solitaire). Each position in $$T$$ represents a pile of cards, and we place each new card on the leftmost pile whose top card is greater than or equal to the new card, or start a new pile if no such pile exists.
