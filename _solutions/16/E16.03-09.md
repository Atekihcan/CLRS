> Show that no compression scheme can expect to compress a file of randomly chosen 8-bit characters by even a single bit. (Hint: Compare the number of possible files with the number of possible encoded files.)

This is a beautiful application of a counting argument that reveals a fundamental limit of compression.

Consider files of length $$n$$ bytes, where each byte is a uniformly random 8-bit character. There are $$256^n = 2^{8n}$$ possible such files, and each is equally likely.

Now suppose we have a compression scheme that maps files to compressed representations. For the scheme to achieve any compression at all, the expected length of the compressed output must be strictly less than $$8n$$ bits.

A compressed output of length $$k$$ bits can represent at most $$2^k$$ distinct values. The total number of possible compressed outputs of length strictly less than $$8n$$ bits is:

$$\sum_{k=0}^{8n-1} 2^k = 2^{8n} - 1$$

{% include ads.html %}

Here's the key observation: we have $$2^{8n}$$ possible input files, but only $$2^{8n} - 1$$ possible "shorter" outputs. By the pigeonhole principle, we cannot give every input file a unique compressed representation that is shorter than $$8n$$ bits.

In fact, any lossless compression scheme must be a one-to-one mapping (otherwise we couldn't decompress uniquely). Since we have $$2^{8n}$$ inputs, we need $$2^{8n}$$ distinct outputs. If all outputs were strictly shorter than $$8n$$ bits, we'd only have $$2^{8n} - 1$$ possible outputs, which is one too few.

Therefore, at least one input file must map to an output of length $$\ge 8n$$ bits. When files are chosen uniformly at random, this "incompressible" file occurs with positive probability.

### Expected Length Analysis

Let's compute the expected compressed length. Let $$L(f)$$ denote the length of the compressed version of file $$f$$. For a uniformly random file $$F$$:

$$\mathbb{E}[L(F)] = \frac{1}{2^{8n}} \sum_{\text{all files } f} L(f)$$

For the compression to be one-to-one, the outputs must be $$2^{8n}$$ distinct binary strings. The sum of lengths of any $$2^{8n}$$ distinct binary strings is minimized when we use all strings of lengths $$0, 1, 2, \ldots$$ in order. But even the first $$2^{8n}$$ strings (in lexicographic order by length) include all strings up to length $$8n - 1$$ plus some strings of length $$8n$$.

More precisely:

$$\sum_{k=0}^{8n-1} 2^k \cdot k + (\text{remaining strings of length } 8n)$$

The expected length cannot be less than $$8n$$ because we cannot fit $$2^{8n}$$ distinct codewords all with length less than $$8n$$ bits.

In fact, by a more careful counting argument, the expected compressed length for random data is at least $$8n$$ bits, meaning we save nothing on average. Some files will compress, but others will expand, and for uniformly random data, these effects cancel out.

{% capture note %}
This result is why practical compression algorithms like gzip, zip, or PNG don't try to compress already-compressed or random data. They often include a "stored" mode that just copies the data when compression would be counterproductive. The entropy of uniformly random 8-bit data is exactly 8 bits per character, which is the fundamental limit that no compression can beat.
{% endcapture %}
