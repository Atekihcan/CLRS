> ***Scheduling variations***
>
> Consider the following algorithm for the problem from Section 16.5 of scheduling unit-time tasks with deadlines and penalties. Let all $$n$$ time slots be initially empty, where time slot $$i$$ is the unit-length slot of time that finishes at time $$i$$. We consider the tasks in order of monotonically decreasing penalty. When considering task $$a_j$$, if there exists a time slot at or before $$a_j$$'s deadline $$d_j$$ that is still empty, assign $$a_j$$ to the latest such slot, filling it. If there is no such slot, assign task $$a_j$$ to the latest of the as yet unfilled slots.
>
> 1. Argue that this algorithm always gives an optimal answer.
> 2. Use the fast disjoint-set forest presented in Section 21.3 to implement the algorithm efficiently. Assume that the set of input tasks has already been sorted into monotonically decreasing order by penalty. Analyze the running time of your implementation.

The scheduling problem in Section 16.5 seeks to minimize total penalty by choosing which tasks to complete on time. This problem presents an alternative algorithm that directly constructs a schedule by assigning tasks to time slots, and asks us to prove its correctness and implement it efficiently.

### A

The algorithm produces an optimal answer because it is equivalent to the greedy matroid algorithm from Section 16.5, just with a different implementation strategy.

First, observe that the algorithm processes tasks in decreasing order of penalty, exactly like the greedy algorithm. The key insight is that assigning a task to the latest available slot at or before its deadline is a correct way to test and maintain independence.

A set of early tasks is independent (achievable without any being late) if and only if they can be scheduled in order of monotonically increasing deadlines without any task missing its deadline. The algorithm's slot-assignment strategy maintains this invariant.

When we consider task $$a_j$$ with deadline $$d_j$$:

- If there's an empty slot at or before $$d_j$$, we can make $$a_j$$ early. Assigning it to the **latest** such slot is crucial: it leaves earlier slots available for tasks with earlier deadlines that we might see later. Since we process by decreasing penalty (not by deadline), a later task might have an earlier deadline.

- If no slot at or before $$d_j$$ is available, task $$a_j$$ must be late. We assign it to the latest unfilled slot overall (which must be after time $$d_j$$), keeping it out of the way of other tasks.

To prove optimality formally, we show this algorithm produces the same set of early tasks as the greedy algorithm from Section 16.5:

1. **Same selection criterion**: Both algorithms select tasks in decreasing penalty order and add them to the early set if and only if they maintain independence.

2. **Correct independence test**: The slot-assignment tests independence correctly. A task can be added to the early set if and only if there's a valid slot for it at or before its deadline. By always choosing the latest available slot, we never "waste" an early slot that a higher-priority task would have needed (since higher-priority tasks were processed first).

3. **Greedy choice preserved**: Each task with the highest remaining penalty is added to the early set whenever possible, matching the greedy algorithm's behavior.

Since the algorithm is equivalent to the optimal greedy algorithm from Theorem 16.11, it produces an optimal schedule.

### B

We use disjoint-set forests to efficiently find the latest available slot at or before a given deadline. The key idea is to group all filled slots together with the latest empty slot that precedes them.

Create $$n + 1$$ sets initially, one for each slot $$0, 1, 2, \ldots, n$$. Slot 0 is a sentinel representing "no valid slot exists." Each set's representative will be the latest empty slot in that contiguous group of slots.

{% capture code %}
Schedule-Tasks(a, d, w, n)
    // Assume tasks are already sorted by decreasing penalty
    // d[j] = deadline of task j, 1 <= d[j] <= n
    let slot[1..n] be a new array initialized to 0
    // slot[i] = task assigned to slot i, 0 means empty

    // Initialize disjoint sets for slots 0 to n
    for i = 0 to n
        Make-Set(i)

    for j = 1 to n
        // Find latest empty slot at or before deadline d[j]
        s = Find-Set(min(d[j], n))
        if s != 0
            // Slot s is available and at or before deadline
            slot[s] = j
            // Merge slot s with slot s - 1
            Union(s - 1, s)
        else
            // No slot available before deadline, task is late
            // Find latest empty slot overall
            s = Find-Set(n)
            if s != 0
                slot[s] = j
                Union(s - 1, s)
            // If s == 0, all slots are filled (shouldn't happen with n tasks and n slots)

    return slot
{% endcapture %}

The $$\textsc{Find-Set}$$ operation returns the representative of the set containing a slot, which is always the latest empty slot in that region. When we fill slot $$s$$, we union it with slot $$s - 1$$, so future queries for any filled slot will return the next empty slot to the left.

{% include ads.html %}

There's one subtlety: after filling slot $$s$$, we need $$\textsc{Find-Set}(s)$$ to return the latest empty slot to its left. By using $$\textsc{Union}(s-1, s)$$ and ensuring the representative of the merged set is $$\textsc{Find-Set}(s-1)$$, we achieve this. The implementation needs a small modification to the union operation: when merging sets containing slots $$s-1$$ and $$s$$, the new representative should be the representative of the set containing $$s-1$$ (i.e., the next empty slot to the left).

{% capture code %}
Union-By-Slot(s_minus_1, s)
    x = Find-Set(s_minus_1)
    y = Find-Set(s)
    if rank[x] > rank[y]
        parent[y] = x
    else if rank[x] < rank[y]
        parent[y] = x    // Force x as representative
        rank[x] = rank[y] + 1
    else
        parent[y] = x
        rank[x] = rank[x] + 1
{% endcapture %}

Alternatively, we can use union by rank normally but keep track of the minimum slot number in each set, updating it during unions.

**Running Time Analysis:**

- Initial setup: $$n + 1$$ calls to $$\textsc{Make-Set}$$, taking $$O(n)$$ time.
- Main loop: $$n$$ iterations, each with at most 2 $$\textsc{Find-Set}$$ operations and 1 $$\textsc{Union}$$ operation.
- Total disjoint-set operations: $$O(n)$$ $$\textsc{Make-Set}$$, $$O(n)$$ $$\textsc{Find-Set}$$, and $$O(n)$$ $$\textsc{Union}$$.

Using the disjoint-set forest with union by rank and path compression, $$m$$ operations on $$n$$ elements take $$O(m \cdot \alpha(n))$$ time, where $$\alpha$$ is the inverse Ackermann function.

Therefore, the total running time is $$O(n \cdot \alpha(n))$$, which is essentially linear. Since the tasks are already sorted, the overall algorithm runs in $$O(n \cdot \alpha(n))$$ time, a significant improvement over the $$O(n^2)$$ algorithm from Section 16.5.

If sorting is required, the running time becomes $$O(n \log n + n \cdot \alpha(n)) = O(n \log n)$$.
