> ***Planning a company party***
>
> Professor Stewart is consulting for the president of a corporation that is planning a company party. The company has a hierarchical structure; that is, the supervisor relation forms a tree rooted at the president. The personnel office has ranked each employee with a conviviality rating, which is a real number. In order to make the party fun for all attendees, the president does not want both an employee and his or her immediate supervisor to attend.
>
> Professor Stewart is given the tree that describes the structure of the corporation, using the left-child, right-sibling representation described in Section 10.4. Each node of the tree holds, in addition to the pointers, the name of an employee and that employee's conviviality ranking. Describe an algorithm to make up a guest list that maximizes the sum of the conviviality ratings of the guests. Analyze the running time of your algorithm.

This is a classic tree DP problem, sometimes called the "maximum weighted independent set on a tree." Think of it like planning a party where you can't invite both a boss and their direct report, but you want to maximize the total "fun factor" of your guest list.

### The Key Insight

For each node (employee) in the tree, we need to make a binary decision: invite them or not. The constraint is that if we invite someone, we cannot invite their immediate parent (supervisor).

For each node $$v$$, we define two quantities:
- $$\text{in}[v]$$: maximum total conviviality if we **include** $$v$$ in the guest list
- $$\text{out}[v]$$: maximum total conviviality if we **exclude** $$v$$ from the guest list

### The Recurrence

**If we include node $$v$$:**

We get $$v$$'s conviviality rating, but we cannot include any of $$v$$'s children. So we must take the "out" value for each child:

$$\text{in}[v] = \text{conviviality}[v] + \sum_{c \in \text{children}(v)} \text{out}[c]$$

**If we exclude node $$v$$:**

We don't get $$v$$'s rating, but we're free to include or exclude each child independently. For each child, we pick whichever option gives the higher value:

$$\text{out}[v] = \sum_{c \in \text{children}(v)} \max(\text{in}[c], \text{out}[c])$$

**Base case (leaf nodes):**
- $$\text{in}[v] = \text{conviviality}[v]$$
- $$\text{out}[v] = 0$$

### The Algorithm

We process the tree from bottom to top (leaves first, then internal nodes, finally the root). This is naturally achieved with a post-order traversal.

{% capture code %}
Party-Planning(T)
    // Perform post-order traversal of tree T
    for each node v in post-order
        if v is a leaf
            in[v] = conviviality[v]
            out[v] = 0
        else
            in[v] = conviviality[v]
            out[v] = 0
            child = v.left-child
            while child != NIL
                in[v] = in[v] + out[child]
                out[v] = out[v] + max(in[child], out[child])
                child = child.right-sibling

    return max(in[root], out[root])
{% endcapture %}

### Reconstructing the Guest List

To actually produce the guest list, we can trace back through our decisions:

{% capture code %}
Print-Guest-List(v, parent-included)
    if v == NIL
        return

    if parent-included
        // Parent is invited, so v cannot be invited
        Print-Guest-List(v.left-child, FALSE)
    else
        // Parent not invited, we can choose for v
        if in[v] >= out[v]
            print v.name
            Print-Guest-List(v.left-child, TRUE)
        else
            Print-Guest-List(v.left-child, FALSE)

    // Process siblings
    Print-Guest-List(v.right-sibling, parent-included)
{% endcapture %}

### Running Time Analysis

The algorithm visits each node exactly once during the post-order traversal. At each node, we iterate through its children using the right-sibling pointers. Across the entire tree, each "child visit" corresponds to exactly one sibling-pointer traversal.

If the tree has $$n$$ nodes:
- Each node is visited once: $$O(n)$$
- Each edge is traversed once (via sibling pointers): $$O(n)$$

Total running time: $$\Theta(n)$$

The space requirement is $$O(n)$$ for storing the $$\text{in}$$ and $$\text{out}$$ arrays, plus $$O(h)$$ for the recursion stack where $$h$$ is the height of the tree. In the worst case (a chain), $$h = n$$, so space is $$O(n)$$.

{% capture note %}
This tree DP pattern appears frequently in competitive programming and real applications. Any time you have a tree structure with local constraints (like "can't pick adjacent nodes"), this bottom-up approach with "include/exclude" states is the standard technique. The linear time complexity makes it highly practical for even large hierarchies.
