> ***Scheduling to minimize average completion time***
>
> Suppose you are given a set $$S = \{a_1, a_2, \ldots, a_n\}$$ of tasks, where task $$a_i$$ requires $$p_i$$ units of processing time to complete, once it has started. You have one computer on which to run these tasks, and the computer can run only one task at a time. Let $$c_i$$ be the completion time of task $$a_i$$, that is, the time at which task $$a_i$$ completes processing. Your goal is to minimize the average completion time, that is, to minimize $$(1/n) \sum_{i=1}^{n} c_i$$. For example, suppose there are two tasks, $$a_1$$ and $$a_2$$, with $$p_1 = 3$$ and $$p_2 = 5$$, and consider the schedule in which $$a_2$$ runs first, followed by $$a_1$$. Then $$c_2 = 5$$, $$c_1 = 8$$, and the average completion time is $$(5 + 8)/2 = 6.5$$. If task $$a_1$$ runs first, however, then $$c_1 = 3$$, $$c_2 = 8$$, and the average completion time is $$(3 + 8)/2 = 5.5$$.
>
> 1. Give an algorithm that schedules the tasks so as to minimize the average completion time. Each task must run non-preemptively, that is, once task $$a_i$$ starts, it must run continuously for $$p_i$$ units of time. Prove that your algorithm minimizes the average completion time, and state the running time of your algorithm.
> 2. Suppose now that the tasks are not all available at once. That is, each task cannot start until its release time $$r_i$$. Suppose also that we allow preemption, so that a task can be suspended and restarted at a later time. For example, a task $$a_i$$ with processing time $$p_i = 6$$ and release time $$r_i = 1$$ might start running at time 1 and be preempted at time 4. It might then resume at time 10 but be preempted at time 11, and it might finally resume at time 13 and complete at time 15. Task $$a_i$$ has run for a total of 6 time units, but its running time has been divided into three pieces. In this scenario, $$a_i$$'s completion time is 15. Give an algorithm that schedules the tasks so as to minimize the average completion time in this new scenario. Prove that your algorithm minimizes the average completion time, and state the running time of your algorithm.

This problem demonstrates the power of the "shortest job first" principle in scheduling. The intuition is simple: by completing shorter tasks early, we avoid having many tasks waiting while a long task runs.

### A

The greedy algorithm is $$\textsc{Shortest-Job-First}$$ (SJF): sort all tasks by processing time in non-decreasing order and execute them in that order.

{% capture code %}
SJF-Schedule(S, p)
    sort tasks in S by processing time p in non-decreasing order
    time = 0
    total = 0
    for each task a in sorted order
        time = time + p[a]
        total = total + time    // completion time of task a
    return total / |S|
{% endcapture %}

To understand why this works, consider what happens when we schedule tasks in some order. If task $$a_j$$ is scheduled at position $$j$$ in the schedule (where positions are numbered 1 through $$n$$), its completion time is the sum of processing times of all tasks scheduled before it plus its own processing time. The total completion time across all tasks is:

$$\sum_{i=1}^{n} c_i = \sum_{j=1}^{n} (n - j + 1) \cdot p_{\pi(j)}$$

where $$\pi(j)$$ denotes the task scheduled at position $$j$$. This formula reflects that the task at position $$j$$ contributes its processing time to the completion times of itself and all $$n - j$$ tasks that come after it.

To minimize this sum, we want tasks with larger processing times to have larger position indices (so they get multiplied by smaller coefficients $$n - j + 1$$). Equivalently, we want tasks with smaller processing times to come first. Sorting by processing time in non-decreasing order achieves exactly this.

{% include ads.html %}

We can prove this formally using an exchange argument. Suppose we have an optimal schedule where task $$a_i$$ precedes task $$a_j$$ but $$p_i > p_j$$. If we swap $$a_i$$ and $$a_j$$:

- Tasks before $$a_i$$ and after $$a_j$$ are unaffected
- The completion time of $$a_j$$ decreases by $$p_i - p_j$$ (now it finishes earlier)
- The completion time of $$a_i$$ increases by $$p_j$$ (now it waits for $$a_j$$)
- Net change: $$-(p_i - p_j) + p_j = 2p_j - p_i < 0$$ (since $$p_i > p_j$$)

The total completion time decreases, contradicting optimality. Therefore, any optimal schedule must have tasks sorted by non-decreasing processing time.

The running time is $$O(n \log n)$$ for sorting, plus $$O(n)$$ to compute the schedule, giving $$O(n \log n)$$ overall.

### B

With release times and preemption allowed, we use $$\textsc{Shortest-Remaining-Time-First}$$ (SRTF): at each moment, run the available task with the shortest remaining processing time. When a new task arrives, we preempt the current task if the new task has shorter remaining time.

{% capture code %}
SRTF-Schedule(S, p, r)
    let remaining[1..n] be a copy of p
    let completed[1..n] = false for all tasks
    Q = min-priority queue (keyed by remaining time)
    events = sorted list of release times with associated tasks
    time = 0
    done = 0
    current = nil
    while done < n
        // Add newly released tasks to the queue
        while events is not empty and next event time <= time
            (t, task) = extract from events
            insert task into Q with key remaining[task]
        // Run the task with shortest remaining time
        if Q is not empty
            current = extract-min(Q)
            // Find next event: either current finishes or new task arrives
            next_event = infinity
            if events is not empty
                next_event = time of next event
            run_time = min(remaining[current], next_event - time)
            time = time + run_time
            remaining[current] = remaining[current] - run_time
            if remaining[current] == 0
                completed[current] = true
                done = done + 1
            else
                insert current into Q
        else
            // No task available, advance to next release
            time = time of next event
    return total completion time / n
{% endcapture %}

The proof of optimality uses a similar exchange argument. Consider any time interval $$[t, t']$$ where the processor is running some task. Among all available tasks at time $$t$$, if we're not running the one with shortest remaining time, we can swap: run the shorter-remaining-time task first, then the other.

Suppose at time $$t$$, task $$a_i$$ with remaining time $$\rho_i$$ is running, but task $$a_j$$ is available with $$\rho_j < \rho_i$$. If we switch to running $$a_j$$:

- $$a_j$$ can finish up to $$\rho_j$$ time units earlier
- $$a_i$$ finishes at most $$\rho_j$$ time units later (it runs after $$a_j$$ completes some work)
- Since $$\rho_j < \rho_i$$, the net effect is a decrease in total completion time

By repeatedly applying this exchange, we arrive at the SRTF schedule, proving it optimal.

The running time analysis: we have at most $$O(n)$$ events (releases and completions). Each event involves $$O(\log n)$$ priority queue operations. Processing all events takes $$O(n \log n)$$ time. Including the initial sort of release times, the total running time is $$O(n \log n)$$.
