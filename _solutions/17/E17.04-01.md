> Suppose that we wish to implement a dynamic, open-address hash table. Why might we consider the table to be full when its load factor reaches some value $$\alpha$$ that is strictly less than 1? Describe briefly how to make insertion into a dynamic, open-address hash table run in such a way that the expected value of the amortized cost per insertion is $$O(1)$$. Why is the expected value of the actual cost per insertion not necessarily $$O(1)$$ for all insertions?

To understand why we might consider an open-address hash table "full" before it actually fills up, we need to think about what happens as an open-address hash table gets crowded. Unlike chained hashing where collisions just add to a list, open-address hashing must probe for an empty slot when a collision occurs. As the table fills up, these probe sequences get longer and longer.

From Theorem 11.6 in Section 11.4, the expected number of probes for an insertion in an open-address hash table with load factor $$\alpha$$ is at most $$1/(1 - \alpha)$$. When $$\alpha = 0.5$$, this is just 2 probes on average. When $$\alpha = 0.9$$, it jumps to 10 probes. When $$\alpha = 0.99$$, it's 100 probes! As $$\alpha$$ approaches 1, the expected probe count goes to infinity.

This is why we define the table to be "full" when the load factor reaches some constant $$\alpha$$ strictly less than 1, say $$\alpha = 0.75$$ or $$\alpha = 0.8$$. By capping the load factor, we guarantee that the expected cost of each insertion (before any expansion) remains bounded by the constant $$1/(1 - \alpha)$$.

To achieve $$O(1)$$ expected amortized cost per insertion, we use the same doubling strategy from Section 17.4.1. When the load factor reaches our threshold $$\alpha$$, we allocate a new table with twice the size and rehash all elements into the new table. Using the potential method analysis from the chapter, each insertion pays for itself plus builds up credit to pay for the next expansion.

Specifically, if we expand when the load factor reaches $$\alpha$$, then after expansion the new load factor is $$\alpha/2$$. The table doubles in size when we insert $$\alpha m - \alpha m/2 = \alpha m/2$$ more elements (where $$m$$ is the table size after expansion). Each of these insertions can be charged a constant amount to pay for rehashing all $$\alpha m$$ elements during the next expansion. The amortized analysis gives us $$O(1)$$ expected amortized cost per insertion.

{% capture note %}
The key insight is that the expected cost of a single insertion depends on the current load factor. Even though most insertions are cheap, an insertion that triggers an expansion has expected cost $$\Theta(n)$$ since we must rehash all $$n$$ elements. The amortized analysis spreads this cost across all the insertions that led to the expansion, giving us $$O(1)$$ expected amortized cost. But the actual cost of that specific expansion-triggering insertion is definitely not $$O(1)$$. It's linear in the table size.

Furthermore, even for non-expansion insertions, the actual cost varies. An insertion into a nearly empty table might find an empty slot immediately (cost 1), while an insertion when the load factor is close to $$\alpha$$ might require $$1/(1-\alpha)$$ probes on average. The $$O(1)$$ bound is an expected amortized bound that averages over both the randomness in the hash function and the sequence of operations.
