> ***Making binary search dynamic***
>
> Binary search of a sorted array takes logarithmic search time, but the time to insert a new element is linear in the size of the array. We can improve the time for insertion by keeping several sorted arrays.
>
> Specifically, suppose that we wish to support $$\textsc{Search}$$ and $$\textsc{Insert}$$ on a set of $$n$$ elements. Let $$k = \lceil \lg(n + 1) \rceil$$, and let the binary representation of $$n$$ be $$\langle n_{k-1}, n_{k-2}, \ldots, n_0 \rangle$$. We have $$k$$ sorted arrays $$A_0, A_1, \ldots, A_{k-1}$$, where for $$i = 0, 1, \ldots, k-1$$, the length of array $$A_i$$ is $$2^i$$. Each array is either full or empty, depending on whether $$n_i = 1$$ or $$n_i = 0$$, respectively. The total number of elements held in all $$k$$ arrays is therefore $$\sum_{i=0}^{k-1} n_i \cdot 2^i = n$$. Although each individual array is sorted, elements in different arrays bear no particular relationship to each other.
>
> 1. Describe how to perform the $$\textsc{Search}$$ operation for this data structure. Analyze its worst-case running time.
>
> 2. Describe how to perform the $$\textsc{Insert}$$ operation. Analyze its worst-case and amortized running times.
>
> 3. Discuss how to implement $$\textsc{Delete}$$.

This data structure cleverly trades off between search and insertion costs. Instead of one large sorted array, we maintain several smaller sorted arrays whose sizes correspond to powers of 2. The connection to binary representation is the key insight: just as adding 1 to a binary number may cause carries that propagate through multiple bit positions, inserting an element may cause merging through multiple arrays.

### A

To search for an element $$x$$, we simply perform binary search on each non-empty array. Since the arrays are independent (elements in different arrays have no ordering relationship), we must check each one.

{% capture code %}
Search(x)
    for i = 0 to k - 1
        if A[i] is not empty
            if Binary-Search(A[i], x) finds x
                return true
    return false
{% endcapture %}
{% include clrs_code.html title="Search(x)" %}

The worst-case analysis: we have at most $$k = \lceil \lg(n+1) \rceil$$ arrays. Binary search on array $$A_i$$ (which has size $$2^i$$) takes $$O(i)$$ time. In the worst case, all arrays are non-empty (when $$n = 2^k - 1$$), and we search each one:

$$\begin{align*}
\sum_{i=0}^{k-1} O(i) &= O\left(\sum_{i=0}^{k-1} i\right) \\
&= O\left(\frac{k(k-1)}{2}\right) \\
&= O(k^2) \\
&= O(\lg^2 n)
\end{align*}$$

So the worst-case search time is $$O(\lg^2 n)$$. This is worse than the $$O(\lg n)$$ for a single sorted array, but the tradeoff enables faster insertions.

### B

Inserting an element is analogous to incrementing a binary number. When we insert, we first create a single-element "array" containing the new element. Then we look at $$A_0$$: if it is empty, we place our new array there. If $$A_0$$ is full, we merge our element with $$A_0$$ (creating a sorted array of size 2), mark $$A_0$$ as empty, and try to place the merged array into $$A_1$$. This process continues until we find an empty slot.

{% capture code %}
Insert(x)
    create a new array B containing just x
    i = 0
    while A[i] is not empty
        B = Merge(A[i], B)
        mark A[i] as empty
        i = i + 1
    A[i] = B
{% endcapture %}
{% include clrs_code.html title="Insert(x)" %}

The $$\textsc{Merge}$$ operation on two sorted arrays of sizes $$m$$ and $$n$$ takes $$\Theta(m + n)$$ time.

For the worst-case analysis, consider what happens when $$n = 2^k - 1$$ (all arrays are full). Inserting one more element causes carries through all $$k$$ positions. We merge arrays of sizes $$1, 2, 4, \ldots, 2^{k-1}$$. The total work is:

$$\begin{align*}
\sum_{i=0}^{k-1} O(2^i) &= O(2^k - 1) \\
&= O(n)
\end{align*}$$

So the worst-case insertion time is $$O(n)$$.

For the amortized analysis, we use an argument similar to the binary counter. Consider how often each array $$A_i$$ participates in a merge. Array $$A_0$$ is involved in every insertion. Array $$A_1$$ is involved in every other insertion. In general, array $$A_i$$ is involved in every $$2^i$$th insertion.

Over $$n$$ insertions, array $$A_i$$ is involved in $$\lfloor n/2^i \rfloor$$ merges, and each merge involving $$A_i$$ costs $$O(2^i)$$ (to merge an array of size $$2^i$$ with the accumulated elements). The total cost is:

$$\begin{align*}
\sum_{i=0}^{k-1} \left\lfloor \frac{n}{2^i} \right\rfloor \cdot O(2^i) &= \sum_{i=0}^{k-1} O(n) \\
&= O(nk) \\
&= O(n \lg n)
\end{align*}$$

Therefore, the amortized cost per insertion is $$O(n \lg n) / n = O(\lg n)$$.

{% capture note %}
We can also use the potential method. Define the potential to be $$\Phi = c \sum_{i=0}^{k-1} i \cdot n_i \cdot 2^i$$, where $$c$$ is a suitable constant. When we insert and merge through $$j$$ arrays (turning bits $$0, 1, \ldots, j-1$$ from 1 to 0 and bit $$j$$ from 0 to 1), the potential change provides credit for the merging work.
{% endcapture %}

### C

Deletion is more challenging because there is no direct analog to "decrementing" in this structure. We have several options:

**Lazy deletion.** Mark elements as deleted without physically removing them. When searching, skip marked elements. Periodically rebuild the structure when too many elements are marked. This maintains $$O(\lg^2 n)$$ search and $$O(1)$$ delete, but wastes space.

**Exact deletion.** To delete element $$x$$, first find which array $$A_i$$ contains it using $$\textsc{Search}$$. Then we need to handle the fact that $$A_i$$ now has size $$2^i - 1$$ instead of $$2^i$$.

One approach: express $$2^i - 1$$ in binary as $$\underbrace{11\ldots1}_{i \text{ ones}}$$. We can split the remaining $$2^i - 1$$ elements into arrays $$A_0, A_1, \ldots, A_{i-1}$$, each of which may need to be merged with its existing contents.

{% capture code %}
Delete(x)
    find array A[i] containing x
    remove x from A[i], leaving 2^i - 1 elements
    split these elements into sorted arrays of sizes 1, 2, 4, ..., 2^(i-1)
    for j = 0 to i - 1
        let B[j] be the array of size 2^j from the split
        if A[j] is empty
            A[j] = B[j]
        else
            merge B[j] into some temporary and continue propagating
{% endcapture %}
{% include clrs_code.html title="Delete(x) [Sketch]" %}

The worst-case deletion involves finding the element ($$O(\lg^2 n)$$), splitting the array ($$O(2^i) = O(n)$$), and potentially merging with existing arrays ($$O(n)$$ in the worst case).

For the amortized analysis, deletion is similar to insertion but in reverse. Using a potential function argument, we can show that deletions have $$O(\lg n)$$ amortized cost when interleaved with insertions, though the analysis is more involved.

A practical approach is to batch deletions: mark elements as deleted, and when the number of marked elements exceeds some threshold (say, half the total), rebuild the entire structure from scratch. This maintains good amortized bounds while keeping the implementation simpler.
