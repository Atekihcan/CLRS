> Suppose we perform a sequence of stack operations on a stack whose size never exceeds $$k$$. After every $$k$$ operations, we make a copy of the entire stack for backup purposes. Show that the cost of $$n$$ stack operations, including copying the stack, is $$O(n)$$ by assigning suitable amortized costs to the various stack operations.

Think of this like a shared document that gets backed up periodically. Every $$k$$ edits, the system saves a complete copy. Each edit is cheap, but the backup could be expensive if the document is large. The key insight is that we can "prepay" for the upcoming backup with each operation, spreading the cost evenly.

We have a stack whose size never exceeds $$k$$, and after every $$k$$ operations, we copy the entire stack (at most $$k$$ elements). The actual costs are:

- $$\textsc{Push}$$: 1
- $$\textsc{Pop}$$: 1
- $$\textsc{Multipop}(s)$$: $$\min(s, \text{stack size})$$
- Backup copy: at most $$k$$ (since stack size $$\le k$$)

The backup happens every $$k$$ operations, so on average we need to pay $$k/k = 1$$ extra per operation just for backups. But we also need to handle the $$\textsc{Multipop}$$ issue from the standard stack analysis.

Let us assign these amortized costs:

- $$\textsc{Push}$$: 4
- $$\textsc{Pop}$$: 0
- $$\textsc{Multipop}$$: 0

When we push an item, we charge 4 dollars:

1. **1 dollar** pays for the actual push
2. **1 dollar** is placed on the item as credit for its eventual pop (as in the standard accounting method analysis)
3. **2 dollars** are placed in a "backup fund"

For $$\textsc{Pop}$$ and $$\textsc{Multipop}$$, we charge nothing. The pop operations are paid by the credit sitting on each item (placed there when it was pushed).

Now consider the backup that occurs after every $$k$$ operations. In those $$k$$ operations, we contributed at most $$2k$$ dollars to the backup fund (at most $$k$$ pushes, each contributing 2 dollars). The backup costs at most $$k$$ dollars (copying at most $$k$$ elements), so the backup fund can always cover it.

For $$n$$ operations, the total amortized cost is at most $$4n$$ (if every operation were a push), which is $$O(n)$$. Since the credit is always nonnegative (we only use credit when popping or backing up, and we've always prepaid), the total amortized cost is an upper bound on the total actual cost.

{% capture note %}
The amortized cost of 4 per push might seem high, but it handles both the pop prepayment and the periodic backup. The key is that we're spreading the backup cost across all operations that precede it, rather than charging it all at once.
{% endcapture %}
