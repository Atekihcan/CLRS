> ***Signing free-agent baseball players***
>
> Suppose that you are the general manager for a major-league baseball team. During the off-season, you need to sign some free-agent players for your team. The team owner has given you a budget of $$\$X$$ to spend on free agents. You are allowed to spend less than $$\$X$$ altogether, but the owner will fire you if you spend any more than $$\$X$$.
>
> You are considering $$N$$ different positions, and for each position, $$P$$ free-agent players who play that position are available. Because you do not want to overload your roster with too many players at any position, for each position you may sign at most one free agent who plays that position. (If you do not sign any players at a particular position, then you plan to stick with the players you already have at that position.)
>
> To determine how valuable a player is going to be, you decide to use a sabermetric statistic known as "VORP," or "value over replacement player." A player with a higher VORP is more valuable than a player with a lower VORP. A player with a higher VORP is not necessarily more expensive to sign than a player with a lower VORP, because factors other than a player's value determine how much it costs to sign him.
>
> For each available free-agent player, you have three pieces of information:
> - the player's position,
> - the amount of money it will cost to sign the player, and
> - the player's VORP.
>
> Devise an algorithm that maximizes the total VORP of the players you sign while spending no more than $$\$X$$ altogether. You may assume that each player signs for a multiple of $100,000. Your algorithm should output the total VORP of the players you sign, the total amount of money you spend, and a list of which players you sign. Analyze the running time and space requirement of your algorithm.

This problem is a variant of the 0/1 knapsack problem with an interesting twist: the items (players) are partitioned into groups (positions), and we can pick at most one item from each group. This is sometimes called the "multiple-choice knapsack problem."

Think of it like shopping for a car with a budget. You need to choose an engine (from several options), a transmission (from several options), a paint color (from several options), and so on. You pick at most one option from each category, trying to maximize total value within your budget.

### Problem Structure

Since player costs are multiples of $100,000, we can normalize: let $$x = X / 100,000$$ be our budget in units. Each player's cost becomes a small integer, making the problem tractable.

Let $$\text{vorp}[i][j]$$ be the VORP of the $$j$$-th player at position $$i$$, and $$\text{cost}[i][j]$$ be their cost (in units of $100,000).

### The Dynamic Programming Solution

Define $$V[i][b]$$ as the maximum total VORP achievable by considering only positions 1 through $$i$$, with a budget of $$b$$ units.

**Base case:** $$V[0][b] = 0$$ for all $$b$$ (no positions considered, no VORP).

**Recurrence:** For position $$i$$, we either:
1. Don't sign anyone at position $$i$$: $$V[i][b] = V[i-1][b]$$
2. Sign player $$j$$ at position $$i$$ (if affordable): $$V[i][b] = V[i-1][b - \text{cost}[i][j]] + \text{vorp}[i][j]$$

$$V[i][b] = \max \left( V[i-1][b], \max_{j: \text{cost}[i][j] \leq b} \left\{ V[i-1][b - \text{cost}[i][j]] + \text{vorp}[i][j] \right\} \right)$$

### The Algorithm

{% capture code %}
Sign-Free-Agents(N, P, vorp, cost, X)
    // Normalize budget to units of $100,000
    x = X / 100000

    // V[i][b] = max VORP using positions 1..i with budget b
    let V[0..N, 0..x] be a new table
    let choice[0..N, 0..x] be a new table  // which player chosen (0 = none)

    // Base case
    for b = 0 to x
        V[0][b] = 0
        choice[0][b] = 0

    // Fill table position by position
    for i = 1 to N
        for b = 0 to x
            // Option 1: don't sign anyone at position i
            V[i][b] = V[i-1][b]
            choice[i][b] = 0

            // Option 2: try each player at position i
            for j = 1 to P
                if cost[i][j] <= b
                    new_vorp = V[i-1][b - cost[i][j]] + vorp[i][j]
                    if new_vorp > V[i][b]
                        V[i][b] = new_vorp
                        choice[i][b] = j

    return V[N][x], choice

### Reconstructing the Solution

{% capture code %}
Print-Signed-Players(choice, cost, vorp, N, x)
    total_vorp = 0
    total_cost = 0
    b = x

    for i = N downto 1
        j = choice[i][b]
        if j != 0
            print "Position" i ": sign player" j
            print "  Cost: $" cost[i][j] * 100000
            print "  VORP:" vorp[i][j]
            total_vorp = total_vorp + vorp[i][j]
            total_cost = total_cost + cost[i][j]
            b = b - cost[i][j]

    print "Total VORP:" total_vorp
    print "Total spent: $" total_cost * 100000

### Complexity Analysis

**Time complexity:** We have $$N \cdot x$$ states, where $$x = X / 100,000$$. For each state, we examine $$P$$ players plus the "sign nobody" option. Total time is $$O(N \cdot P \cdot x) = O(N \cdot P \cdot X / 100,000)$$.

If we denote $$X' = X / 100,000$$ as the normalized budget, the running time is $$O(N \cdot P \cdot X')$$.

**Space complexity:** The DP table requires $$O(N \cdot X')$$ space. We can reduce this to $$O(X')$$ by noting that row $$i$$ only depends on row $$i-1$$, but then we need additional space for reconstruction.

### Example

Suppose we have 3 positions, 2 players per position, and a $500,000 budget ($$x = 5$$ units):

| Position | Player | Cost (units) | VORP |
|----------|--------|--------------|------|
| 1 | 1 | 2 | 15 |
| 1 | 2 | 3 | 20 |
| 2 | 1 | 1 | 8 |
| 2 | 2 | 4 | 25 |
| 3 | 1 | 2 | 12 |
| 3 | 2 | 3 | 18 |

With budget 5, one optimal solution is:
- Position 1: Player 1 (cost 2, VORP 15)
- Position 2: Player 1 (cost 1, VORP 8)
- Position 3: Player 2 (cost 3... wait, that's 6 total, over budget)

Let's recalculate:
- Position 1: Player 1 (cost 2, VORP 15)
- Position 2: Player 1 (cost 1, VORP 8)
- Position 3: Player 1 (cost 2, VORP 12)
- Total: cost 5, VORP 35

The algorithm systematically explores all valid combinations to find the maximum VORP within budget.

{% capture note %}
This problem illustrates how real-world constraints (budget, one player per position) map naturally to dynamic programming state spaces. The "multiple of $100,000" constraint is crucial: it discretizes the budget into a manageable number of states. Without it, we'd need to handle continuous costs, making the problem much harder.
