> ***Bit-reversed binary counter***
>
> Chapter 30 examines an important algorithm called the fast Fourier transform, or FFT. The first step of the FFT algorithm performs a bit-reversal permutation on an input array $$A[0 \ldots n-1]$$ whose length is $$n = 2^k$$ for some nonnegative integer $$k$$. This permutation swaps elements whose indices have binary representations that are the reverse of each other.
>
> We can express each index $$a$$ as a $$k$$-bit sequence $$\langle a_{k-1}, a_{k-2}, \ldots, a_0 \rangle$$, where $$a = \sum_{i=0}^{k-1} a_i 2^i$$. We define
>
> $$\text{rev}_k(\langle a_{k-1}, a_{k-2}, \ldots, a_0 \rangle) = \langle a_0, a_1, \ldots, a_{k-1} \rangle$$
>
> thus, $$\text{rev}_k(a) = \sum_{i=0}^{k-1} a_{k-i-1} 2^i$$.
>
> For example, if $$n = 16$$ (or, equivalently, $$k = 4$$), then $$\text{rev}_k(3) = 12$$, since the 4-bit representation of 3 is 0011, which when reversed gives 1100, the 4-bit representation of 12.
>
> 1. Given a function $$\text{rev}_k$$ that runs in $$\Theta(k)$$ time, write an algorithm to perform the bit-reversal permutation on an array of length $$n = 2^k$$ in $$O(nk)$$ time.
>
> We can use an algorithm based on an amortized analysis to improve the running time of the bit-reversal permutation. We maintain a "bit-reversed counter" and a procedure $$\textsc{Bit-Reversed-Increment}$$ that, when given a bit-reversed-counter value $$a$$, produces $$\text{rev}_k(\text{rev}_k(a) + 1)$$. If $$k = 4$$, for example, and the bit-reversed counter starts at 0, then successive calls to $$\textsc{Bit-Reversed-Increment}$$ produce the sequence $$0000, 1000, 0100, 1100, 0010, 1010, \ldots = 0, 8, 4, 12, 2, 10, \ldots$$.
>
> 2. Assume that the words in your computer store $$k$$-bit values and that in unit time, your computer can manipulate the binary values with operations such as shifting left or right by arbitrary amounts, bitwise-AND, bitwise-OR, etc. Describe an implementation of the $$\textsc{Bit-Reversed-Increment}$$ procedure that allows the bit-reversal permutation on an $$n$$-element array to be performed in a total of $$O(n)$$ time.
>
> 3. Suppose that you can shift a word left or right by only one bit in unit time. Is it still possible to implement an $$O(n)$$-time bit-reversal permutation?

The bit-reversal permutation is a crucial preprocessing step in the FFT algorithm. Understanding how to compute bit-reversed indices efficiently demonstrates the power of amortized analysis applied to a practical problem.

### A

The straightforward approach is to iterate through all indices from 0 to $$n-1$$, compute the bit-reversed index for each, and swap elements when the reversed index is greater than the current index (to avoid swapping twice).

{% capture code %}
Bit-Reversal-Permutation(A, n)
    k = lg n
    for i = 0 to n - 1
        j = rev_k(i)
        if i < j
            exchange A[i] with A[j]
{% endcapture %}
{% include clrs_code.html title="Bit-Reversal-Permutation(A, n)" %}

The algorithm makes $$n$$ calls to $$\text{rev}_k$$, each costing $$\Theta(k) = \Theta(\lg n)$$ time. The total running time is therefore $$O(nk) = O(n \lg n)$$.

The condition $$i < j$$ ensures we only swap each pair once. We could equivalently check $$i < j$$ because an index $$i$$ and its reversal $$j = \text{rev}_k(i)$$ form a pair, and swapping only when $$i < j$$ handles each pair exactly once.

### B

The key insight is that we can increment a bit-reversed counter efficiently by exploiting the structure of bit reversal. When we increment a normal counter, we flip the rightmost 0 bit to 1 and all the bits to its right (which are 1s) to 0. In a bit-reversed counter, these operations happen from the left side instead.

Think of a normal increment: we scan from the right, flipping 1s to 0s until we find a 0, then flip it to 1. For a bit-reversed counter, we do the same but scanning from the left (the most significant bit).

{% capture code %}
Bit-Reversed-Increment(a, k)
    m = 2^(k-1)
    while a AND m != 0
        a = a XOR m
        m = m / 2
    a = a OR m
    return a
{% endcapture %}
{% include clrs_code.html title="Bit-Reversed-Increment(a, k)" %}

Here $$m$$ starts as $$2^{k-1}$$ (a 1 in the leftmost position) and moves right. We flip 1 bits to 0 (using XOR) until we find a 0 bit, then flip it to 1 (using OR).

With arbitrary shifts available in unit time, we can compute $$m = 2^{k-1}$$ and perform $$m = m/2$$ (which is a right shift by 1) in constant time per iteration.

To analyze the amortized cost, we use the same argument as for the standard binary counter. Each bit position gets flipped at most once per two increments on average. Specifically, the leftmost bit (position $$k-1$$) flips every time, the next bit flips every other time, and so on. The total number of bit flips across $$n$$ increments is:

$$\begin{align*}
\sum_{i=0}^{k-1} \left\lfloor \frac{n}{2^i} \right\rfloor &< n \sum_{i=0}^{\infty} \frac{1}{2^i} \\
&= 2n
\end{align*}$$

Thus $$n$$ calls to $$\textsc{Bit-Reversed-Increment}$$ cost $$O(n)$$ total, giving an amortized cost of $$O(1)$$ per increment. The complete bit-reversal permutation algorithm becomes:

{% capture code %}
Bit-Reversal-Permutation-Fast(A, n)
    k = lg n
    j = 0
    for i = 0 to n - 1
        if i < j
            exchange A[i] with A[j]
        j = Bit-Reversed-Increment(j, k)
{% endcapture %}
{% include clrs_code.html title="Bit-Reversal-Permutation-Fast(A, n)" %}

The total running time is $$O(n)$$ for the loop iterations plus $$O(n)$$ total for all the bit-reversed increments, giving $$O(n)$$ overall.

### C

Yes, it is still possible to achieve $$O(n)$$ time even when we can only shift by one bit at a time.

The key observation is that we do not need to recompute the mask $$m$$ from scratch each time. Instead, we can maintain the current mask position and update it incrementally.

When we call $$\textsc{Bit-Reversed-Increment}$$, we start with a mask at the leftmost position. As we scan through 1 bits (flipping them to 0), we move the mask right. When we finally find a 0 bit and flip it to 1, we remember where the mask ended up. On the next call, instead of starting from the leftmost position, we can use information about where the previous call ended.

Alternatively, we can observe that the mask position after processing index $$i$$ tells us something about the structure of the bit-reversed counter value. We maintain the mask $$m$$ as a global variable, and the amortized analysis still applies.

More concretely, consider maintaining both the bit-reversed counter value $$a$$ and the current mask $$m$$. When incrementing:

1. While the bit at position $$m$$ is 1, flip it to 0 and move $$m$$ right (one shift)
2. When we find a 0 bit, flip it to 1
3. Reset $$m$$ back to $$2^{k-1}$$ only when needed

The total number of mask movements across all $$n$$ increments is still $$O(n)$$ by the same counting argument. Each left shift to reset the mask can be charged to the right shifts that preceded it.

{% capture note %}
Another approach is to notice that the number of times the mask must be shifted left to reset to position $$k-1$$ equals the number of times it was shifted right. Since the total right shifts over $$n$$ increments is $$O(n)$$ (by the amortized analysis), the total left shifts is also $$O(n)$$. Therefore, even with unit-cost single-bit shifts, the total time remains $$O(n)$$.
{% endcapture %}
{% include aside.html title='Why this still works' %}

The fundamental insight is that amortized analysis counts the total work, not the worst-case work per operation. Whether we shift by many bits at once or one bit at a time, the total number of bit positions visited across all operations remains the same.
