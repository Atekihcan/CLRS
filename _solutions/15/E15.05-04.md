> Knuth [212] has shown that there are always roots of optimal subtrees such that $$root[i, j - 1] \le root[i, j] \le root[i + 1, j]$$ for all $$1 \le i < j \le n$$. Use this fact to modify the $$\textsc{Optimal-BST}$$ procedure to run in $$\Theta(n^2)$$ time.

This is a beautiful example of how a mathematical insight about the structure of optimal solutions can dramatically improve algorithm efficiency. Knuth's result tells us that the optimal root for a range $$[i, j]$$ is "sandwiched" between the optimal roots of two slightly smaller ranges: $$[i, j-1]$$ (same left endpoint, shorter by one on the right) and $$[i+1, j]$$ (shorter by one on the left, same right endpoint).

Think of it this way: as we expand a range by adding one more key, the optimal root can only shift in a predictable direction. It cannot suddenly jump to a very different location. This monotonicity property allows us to search far fewer candidates for each root.

In the original $$\textsc{Optimal-BST}$$, the innermost loop (lines 10-14) tries all possible roots $$r$$ from $$i$$ to $$j$$, which takes $$O(j - i)$$ time per subproblem. The key insight is that Knuth's property lets us restrict this search range.

Instead of searching $$r$$ from $$i$$ to $$j$$, we can search from $$root[i, j-1]$$ to $$root[i+1, j]$$. Since these bounds come from previously computed entries in the $$root$$ table (computed when we processed smaller subproblems), this information is available when we need it.

{% capture code %}
Optimal-BST-Knuth(p, q, n)
    let e[1..n+1, 0..n], w[1..n+1, 0..n],
        and root[1..n, 1..n] be new tables
    // Initialize base cases
    for i = 1 to n + 1
        e[i, i - 1] = q[i-1]
        w[i, i - 1] = q[i-1]
    // Initialize root for single-key subtrees
    for i = 1 to n
        root[i, i] = i
    // Fill in tables for increasing subproblem sizes
    for l = 1 to n
        for i = 1 to n - l + 1
            j = i + l - 1
            e[i, j] = infinity
            w[i, j] = w[i, j - 1] + p[j] + q[j]
            // Knuth's optimization: restrict search range
            if l == 1
                r_low = i
                r_high = j
            else
                r_low = root[i, j - 1]
                r_high = root[i + 1, j]
            for r = r_low to r_high
                t = e[i, r - 1] + e[r + 1, j] + w[i, j]
                if t < e[i, j]
                    e[i, j] = t
                    root[i, j] = r
    return e and root
{% endcapture %}

**Why does this achieve $$\Theta(n^2)$$ time?**

The key is to analyze how many root candidates we examine in total across all subproblems. Let's count the total number of iterations of the innermost loop.

For a fixed subproblem length $$l \ge 2$$, consider all subproblems $$[i, i+l-1]$$ for $$i = 1, \ldots, n-l+1$$. The innermost loop for subproblem $$[i, j]$$ where $$j = i + l - 1$$ runs from $$root[i, j-1]$$ to $$root[i+1, j]$$.

The total number of iterations for all subproblems of length $$l$$ is:

$$\sum_{i=1}^{n-l+1} (root[i+1, j] - root[i, j-1] + 1)$$

This sum telescopes! Let's see why. Writing out the terms:

$$\begin{align*}
&(root[2, l] - root[1, l-1] + 1) \\
+\, &(root[3, l+1] - root[2, l] + 1) \\
+\, &(root[4, l+2] - root[3, l+1] + 1) \\
&\vdots \\
+\, &(root[n-l+2, n] - root[n-l+1, n-1] + 1)
\end{align*}$$

Most terms cancel (telescope), leaving:

$$root[n-l+2, n] - root[1, l-1] + (n - l + 1)$$

Since $$root$$ values are bounded between 1 and $$n$$, this sum is at most:

$$n - 1 + (n - l + 1) = 2n - l \le 2n$$

So for each length $$l$$, the total work across all subproblems is $$O(n)$$. Summing over all $$n$$ lengths:

$$\sum_{l=1}^{n} O(n) = O(n^2)$$

The base case ($$l = 1$$) contributes $$O(n)$$ work since each single-key subtree has exactly one candidate root. The $$w$$ table still takes $$O(1)$$ per entry, totaling $$O(n^2)$$. Therefore, the overall running time is $$\Theta(n^2)$$.

{% capture note %}
Knuth's optimization is an example of a general technique sometimes called the "monotonicity optimization" or "divide-and-conquer optimization" for dynamic programming. The key insight is that when optimal choices have a monotonicity property (i.e., they don't decrease or increase too wildly as parameters change), we can exploit this to reduce the search space. This technique applies to various other DP problems beyond optimal BSTs.
{% endcapture %}
{% include aside.html title='The Bigger Picture' %}

This improvement from $$\Theta(n^3)$$ to $$\Theta(n^2)$$ is significant. For $$n = 1000$$ keys, the original algorithm might require roughly a billion operations, while the optimized version requires only about a million. This could mean the difference between a program running in hours versus seconds.
